Esta es una situación clásica en sistemas que tienen "doble entrada" (Registro Web vs. Registro Presencial). Tienes toda la razón: el recepcionista NO debe inventar ni saber la contraseña del usuario, y pedirle al cliente que la escriba en el teclado de recepción es incómodo e inseguro.

Aquí te presento la solución estándar de la industria para este caso: El Flujo de "Cuenta Pendiente de Activación".

La Estrategia: "Registro Rápido + Activación en Casa"
El flujo funciona así:

En Recepción (El problema que tienes): El empleado registra SOLO los datos vitales (DNI, Nombre, Apellido, Plan).

¿Y el Usuario? El sistema asigna automáticamente el DNI como NombreUsuario.

¿Y la Contraseña? Se guarda como NULL en la base de datos.

En Casa (La solución): El cliente intenta entrar a la Web.

Opción A: Intenta registrarse de nuevo -> El sistema le dice: "Ya eres socio, activa tu cuenta aquí".

Opción B: Va a un link de "Primer Ingreso / Activar Cuenta".

Activación: El usuario valida su identidad (con su DNI y quizás Email) y es ahí donde ÉL MISMO crea su contraseña y personaliza su nombre de usuario si quiere.

Implementación Técnica
Para lograr esto, necesitamos ajustar ligeramente tu lógica en el UsuariosController y en el UsuarioService.

1. Ajuste en el Modelo (ViewModel)
Tu UsuarioViewModel actualmente obliga a poner contraseña siempre ([Required]). Debemos hacerlo condicional o crear un ViewModel separado para administración.

Para no complicarnos, usaremos lógica en el Controlador: La contraseña es obligatoria SOLO si el UserId es 0 (Crear) Y el registro se hace desde la Web Pública. Si lo hace el Admin, es opcional.

2. Lógica en UsuarioService (Manejo de Contraseña Nula)
Modifica tu método CrearUsuarioAsync para soportar la creación sin contraseña (modo Recepción).

C#

public async Task<Usuario> CrearUsuarioAsync(Usuario usuario, string passwordRaw)
{
    // ... validaciones de DNI y Usuario ...

    // CAMBIO CLAVE:
    if (string.IsNullOrEmpty(passwordRaw)) 
    {
        // Si no hay contraseña (Registro por Recepción), dejamos el Hash en NULL
        usuario.PasswordHash = null;
        
        // Regla de Negocio: Si no definieron usuario, usamos el DNI
        if (string.IsNullOrEmpty(usuario.NombreUsuario))
        {
            usuario.NombreUsuario = usuario.Dni;
        }
    }
    else 
    {
        // Si hay contraseña (Registro Web), hasheamos normal
        usuario.PasswordHash = BCrypt.Net.BCrypt.HashPassword(passwordRaw);
    }

    // ... resto del código (QR, fechas, guardar) ...
}

4. Cómo el usuario "Activa" su cuenta (Nuevo Endpoint)
Cuando el usuario registrado en recepción quiera entrar a la web, no podrá porque su password es NULL. Necesitas una vista simple de "Activar Cuenta".

[HttpPost]
public async Task<IActionResult> ActivarCuenta(string dni, string email, string newPassword)
{
    var usuario = await _usuarioRepository.ObtenerPorDNIAsync(dni);

    if (usuario == null) 
        return View("Error", "Usuario no encontrado");

    // VALIDACIÓN DE SEGURIDAD: 
    // Verificar que el email coincida con el que dio en recepción
    // Opcional: Si no dio email en recepción, pedirle otros datos (Fecha Nacimiento, etc.)
    if (usuario.Email != email) 
        return View("Error", "El email no coincide con nuestros registros.");

    if (usuario.PasswordHash != null)
        return View("Error", "Esta cuenta ya está activa. Intente iniciar sesión.");

    // TODO OK: Creamos la contraseña
    usuario.PasswordHash = BCrypt.Net.BCrypt.HashPassword(newPassword);
    await _usuarioRepository.UpdateAsync(usuario);
    await _usuarioRepository.SaveAsync();

    return RedirectToAction("Login", new { mensaje = "Cuenta activada. Ingresa ahora." });
}
Resumen de la Solución
En Recepción: El formulario de "Nuevo Cliente" oculta los campos de contraseña. El sistema guarda PasswordHash = NULL y NombreUsuario = DNI.

El Cliente: Se va con su membresía activa (QR funcionando) pero sin acceso web.

En Web: Cuando quiera entrar, el sistema detecta que no tiene password y le pide crear una ("Activar Cuenta") validando su DNI y Email.
